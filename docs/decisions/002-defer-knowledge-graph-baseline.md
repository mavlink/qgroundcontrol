# ADR-002 — Defer the `/understand-anything` knowledge graph baseline

**Status:** Accepted, 2026-05-12.
**Owner:** stratten.

## Context

Phase 0 kickoff asked for a full knowledge graph generated by `/understand-anything` over the QGC tree, committed under `docs/knowledge-graph/baseline/`. The repo is ~1,945 C++/QML source files plus translations, resources, tooling, and tests. The skill's own design envelope warns at >100 files; QGC is ~20× that.

A full run would take many hours, cost significant inference budget, and produce an artifact whose primary consumers (architecture tours, onboarding docs) are better served by a focused human-written overview at this stage of the fork.

## Decision

Skip the baseline graph for now. Instead, deliver Phase 0's architectural understanding through:

1. `docs/architecture/00-overview.md` — synthesized from six focused subsystem surveys (MissionManager, MAVLink/Vehicle/Comms, FirmwarePlugin/AutoPilotPlugin, custom build system, QML layering, VideoManager).
2. `UPSTREAM_PATCHES.md` ledger with anticipated patches and reasoning.
3. ADR-001 (this directory) for the fork decision itself.

Generate the graph later, scoped per-subsystem, once Phase 1 (branding) is in. A per-subsystem graph (~100–300 files each) fits the skill's design envelope and stays useful for incremental updates as Sprig diverges from upstream.

## Consequences

**Accepted because:**

- The Phase 0 deliverable the kickoff actually needed is *understanding*, not an automated artifact. The six subsystem surveys (run as parallel `Explore` subagents) provided the cited-file:line specifics that an architecture tour needs.
- A whole-repo graph for a Qt codebase this size is noisy — half of it is translations, resources, and platform packaging that the tours filter out anyway.
- Per-subsystem graphs after Phase 1 will be cheaper to maintain (`knowledge-graph` skill has incremental update support) and align with the kickoff's six named tours.

**Costs accepted:**

- No machine-readable graph for the dashboard / chat / explain tools that the `understand-anything` ecosystem provides. We give up that workflow until the per-subsystem run.
- Future contributors can't lean on the dashboard to orient themselves; they read `00-overview.md` instead. The overview is sized to be readable in one sitting.

**Revisit when:**

- Phase 1 lands and we want to scope per-subsystem graphs (`MissionManager/`, `Vehicle/+Comms/+MAVLink/`, `FirmwarePlugin/+AutoPilotPlugins/`, `UI/+QmlControls/+FlyView/+PlanView/`, `VideoManager/`, `custom/`).
- A contributor specifically asks for the dashboard to navigate a code-review.
